## 1. 设计目标

### 1.1 核心问题
游戏/心理学/用户研究领域的访谈分析面临三个核心痛点：

1. **转写耗时** — 研究者花费大量时间将录音转为文字
2. **整理繁琐** — 从大段对话中手动提取受访者回答并匹配到结构化问题
3. **编码不统一** — 质性研究编码（扎根理论）缺乏一致性，不同研究者编码结果差异大

### 1.2 设计原则

| 原则 | 说明 |
|------|------|
| **LLM-First** | 尽可能利用 LLM 的语义理解能力，减少规则代码 |
| **模块化** | ASR、LLM分析、报告生成各自解耦，可独立替换 |
| **低门槛** | 纯 Python 知识库、多 ASR 方案，降低部署和使用成本 |
| **双模式输出** | 同一数据同时输出「原话版」和「编码版」 |

### 2.2 数据流

```
音频文件 (.m4a/.mp3/.wav)
    │
    ├─ ffmpeg → 16kHz/mono/16bit WAV（标准化预处理）
    │
    ├─ ASR 引擎
    │   ├─ OpenAI Whisper API（云端，需 API Key）
    │   └─ 阿里云 DashScope Paraformer（云端，需阿里云配置）
    │
    ├─ 转写文本
    │
    ├─ LLM 语义分析（DeepSeek/OpenAI 兼容 API）
    │   ├─ 提取受访者回答片段
    │   ├─ 匹配到结构化问题
    │   ├─ 知识库增强（RAG）
    │   └─ 主题编码 + 情感分析
    │
    └─ 双版本报告（.xlsx）
        ├─ 原话版：问题 ↔ 回答 1:1 映射
        └─ 编码版：含主题编码、子编码、关键词、情感倾向
```

## 3. 核心模块设计

### 3.1 ASR 模块 (`utils/asr.py`)

**职责**: 将音频文件转为文本

**支持的引擎**:

| 引擎 | 类型 | 优点 | 缺点 |
|------|------|------|------|
| OpenAI Whisper API | 云端API | 精度高，配置简单 | 需要 OpenAI Key |
| 阿里云 DashScope | 云端API | 中文优化，国内速度快 | 需要 OSS + DashScope 配置 |

**关键技术决策**:
- 使用阿里云 `transcription-async` 异步接口（同步接口需要特定权限）
- OSS 使用 HTTPS 协议连接，兼容外部平台网络环境
- 音频预处理标准化：16kHz 采样率、单声道、16bit 位深

### 3.2 LLM 分析引擎 (`utils/llm_utils.py`)

**职责**: 语义匹配、主题编码、情感分析

**能力**:
- `extract_answers_with_kb()` — LLM 从长文本中提取回答并匹配到问题
- `generate_thematic_coding()` — 生成主题编码（一级编码 + 二级编码）
- `analyze_sentiment()` — 情感倾向分析

**Prompt 设计策略**:
- 结构化的 System Prompt，明确角色、任务、输出格式
- 使用 Few-shot 示例提升编码一致性
- 通过 JSON Schema 约束输出格式

### 3.3 记忆管理 (`utils/memory.py`)

**职责**: 存储和管理问答对

**数据结构**:
```
responses: dict[str, list[dict]]  — question_id → [回答列表]
unmatched: list[dict]              — 无法匹配的片段
fuzzy_match: list[dict]            — 低置信度匹配
```

**全局单例模式**: 通过 `get_memory()` / `reset_memory()` 管理全局状态

### 3.4 知识库 (`utils/knowledge_base.py`)

**职责**: 轻量级 RAG，存储参考文档并检索相关上下文

**设计决策**:
- **为什么不用向量数据库？** — 降低部署门槛，零外部依赖
- **检索策略**: 关键词匹配 + LLM 重排
- **分块策略**: 长文本自动切分，每块约 500 字符

**数据结构**:
```python
documents: list[dict] = [
    {"id": str, "text": str, "category": str,
     "title": str, "source": str}
]
```

### 3.5 报告生成 (`utils/report.py`)

**职责**: 生成双版本 Excel 报告

**两个版本**:

| 维度 | 原话版 | 编码版 |
|------|--------|--------|
| 内容 | 问题 ↔ 回答映射 | 原话版 + 主题编码 + 情感 |
| 列数 | 5列 | 9列 |
| 用途 | 原始数据存档 | 分析报告 |
| 示例 | Q1 → "我喜欢玩原神" | Q1 → RPG/开放世界/正面 |

**样式**:
- 蓝色表头，隔行黄色标记
- 自适应列宽，自动换行
- 条件格式（置信度 < 0.5 黄色高亮）
- UTF-8 BOM 编码，Excel 直接打开不乱码

## 4. 关键技术决策

### 4.1 为什么选择 LangGraph 而非直接调用 LLM？
- Agent 需要管理多步骤流程（ASR → 分析 → 报告）
- 需要记忆能力追踪对话状态
- 工具调用机制让 LLM 决定何时使用外部能力

### 4.2 为什么选择多 ASR 方案？
- OpenAI Whisper 精度高但需要海外 API
- 阿里云 DashScope 中文优化好但需要国内云资源配置
- 双方案让用户根据自身条件选择

### 4.3 为什么选择纯 Python RAG？
- 避免引入向量数据库的部署复杂度
- 对于中小规模知识库（<1000 条），关键词检索 + LLM 重排效果足够
- 用户可零配置使用
