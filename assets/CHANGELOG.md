# Prompt v3 — Changelog & 预期效果

## 一、模板占位符约定

v3 prompts 使用 `{{PLACEHOLDER}}` 双花括号占位符,避免与 JSON 示例中的单花括号冲突。
推荐的加载方式(简单字符串替换,不用 `.format()`,因为 JSON 示例里有单括号):

```python
def render_prompt(template: str, **kwargs) -> str:
    for key, value in kwargs.items():
        template = template.replace(f"{{{{{key}}}}}", str(value))
    return template
```

各 prompt 需要替换的占位符:
- `extract_answers.txt`: `{{QUESTIONS}}`、`{{TRANSCRIPT}}`、`{{KB_CONTEXT_SECTION}}`
- `code_response.txt`: `{{Q_ID}}`、`{{QUESTION_TEXT}}`、`{{ANSWER_TEXT}}`、`{{KB_CONTEXT_SECTION}}`
- `clean_text.txt`: `{{TEXT}}`

`{{KB_CONTEXT_SECTION}}` 在没有 KB 时传空串;有 KB 时按以下格式填:

```
【知识库参考】
以下为过往访谈案例和领域术语,仅作匹配参考,不作为"造话"的依据:
{KB 内容}
```

## 二、v2 → v3 关键变更

### `extract_answers`

| v2 问题 | v3 改动 |
|---|---|
| 无反幻觉约束,LLM 可能输出转写中不存在的 original_text | 新增「绝对原则」第 1 条:不准捏造,逐字片段;并在示例中演示 |
| 无 OFF_TOPIC 出口,跑题内容被强塞到某个题目 | 输出 schema 加 `"question_id":"OFF_TOPIC"`;示例中给出一条 OFF_TOPIC 样本 |
| score 无评分标准,跨调用不一致 | 给出 4 档评分细则(0.85+ / 0.6-0.85 / 0.4-0.6 / <0.4) |
| "1-20 个字"规则误导模型跳过短答 | 改为「可长可短,1 个字也可能有效」,去掉数字框 |
| cleaned_text 让 LLM 自由清洗 | 明确「仅删除填充词,不可润色 / 改写语序 / 合并 / 补全」;示例演示保留有功能的"那个" |
| 无 few-shot 示例 | 加 3 条样本(含一条 OFF_TOPIC) |
| 无 rationale 字段,无法审计为何这样匹配 | 输出加 `rationale` 字段 |

### `code_response`

| v2 问题 | v3 改动 |
|---|---|
| 预设 5 个一级类别变相强制套用,破坏"开放编码" | 类别从「使用这些类」降级为「卡住时启发」,并显式说明先尝试从数据本身产码 |
| 每段只输出一个码,损失多维度信息 | 输出 `candidate_codes` 数组,1–3 条 |
| 无 source_quote 锚点,研究者无法追溯依据 | 每个候选码必带 source_quote,且要求逐字匹配 |
| 无 descriptive / in_vivo 区分,丢失 in-vivo 编码价值 | 加 `type` 字段;示例同时展示两种 |
| sentiment 强制输出,污染纯事实陈述 | 增加 `"n/a"` 选项,明确「仅在带情感色彩时填」 |
| 无人工核验占位字段 | 加 `human_decision` / `human_code` / `note` 三个 null 字段,留给研究者填,作为采纳率指标的载体 |
| 无 confidence 字段 | 加 `confidence`(high/medium/low)+ 评分细则 |
| 失败兜底返回「未分类 / 待确认」,污染下游 | 显式要求「宁可标 low confidence,也不要给敷衍的码」 |

### `clean_text`

| v2 问题 | v3 改动 |
|---|---|
| "保留核心语义"过于宽松,允许 LLM 润色改写 | 改为白名单枚举:只可删除明确列出的填充词类型 |
| 不返回 removals,无 audit 路径 | 输出加 `removals` 数组,记录每个被删词 |
| 不区分"那个"作填充词 vs 作指代词 | 加专门小节说明,并用「示例 3」演示作指代词时保留 |
| 无示例 | 加 3 个示例(含无填充词的 no-op 案例和"那个"作指代词的反例) |

## 三、预期效果(诚实版)

**先声明:我无法给你具体的百分比提升数字,而且任何给你"准确率提升 30%"这类话的人都是在编。** 没有评测集 = 没有数字。下面是基于这些 prompt 改动的**定性预测**——它们应该能改善哪些失败模式、改善到什么程度。

### v2 在你真实数据上的可能失败模式(基于经验估计)

下列是 LLM 在这类任务上的常见失败,顺序大致按发生频率从高到低:

1. **`extract_answers` 中的「准引用幻觉」**:LLM 输出的 original_text 不是转写中的精确连续片段,而是"它觉得受访者说的意思",细节有偏差。v2 没有任何机制约束这一点,**预计在长访谈中相当常见**。
2. **`code_response` 高度集中到 5 个预设类别**:几乎所有码会落入「产品功能评价 / 情感态度 / 使用行为 / 竞品对比 / 推荐意愿」之一,**多样性损失严重**。这不是 LLM 的错,是 prompt 在引导它这么做。
3. **score 在 0.5–0.8 之间剧烈波动**:同一段文本两次调用得到 0.6 和 0.75,因为 v2 没有评分细则。
4. **cleaned_text 过度清洗**:有功能的"那个 / 就是"被一并删掉,语义损失。
5. **跑题内容被硬塞到某个题目**:v2 没有 OFF_TOPIC 出口,LLM 被迫匹配。
6. **`clean_text` 偶发改写**:不只是删除,顺手调整了语序或合并了句子。

### v3 应该改善的方向

| 失败模式 | v3 是否能改善 | 改善机制 |
|---|---|---|
| 准引用幻觉 | **能,显著** | 明确「逐字片段」约束 + 示例演示。建议 + 代码侧加一道校验:`original_text in transcript`,不通过就拒绝。 |
| 类别趋同 | **能,显著** | 类别从「使用」降级为「启发」;in_vivo 类型逼出贴近数据的码 |
| score 不稳 | **能,中等** | 评分细则缩小了 LLM 的解释空间,但仍有波动。彻底解决要把 score 离散化(high/medium/low) |
| 过度清洗 | **能,显著** | 白名单 + 指代词专门说明 |
| 跑题硬塞 | **能,显著** | OFF_TOPIC 选项 + 示例 |
| clean_text 改写 | **能,显著** | 白名单 + removals 审计 |
| 编码可解释性 | **从无到有** | source_quote + rationale + confidence,每个码可追溯 |
| 编码采纳率指标 | **从无到有** | human_decision / human_code 字段,导出 Excel 后由人填 |

### 不会改善的部分(诚实告知)

- **LLM 本身的语义判断错误**:如果模型本身没听懂某段话,prompt 改不动它的能力上限。
- **ASR 错误**:转写错了的字,extract 阶段也救不回来——这是另一个环节的事(术语表纠错)。
- **token 成本 / 延迟**:v3 prompts 更长、输出更结构化,**token 消耗会增加大约 20–40%**,延迟相应增加。如果你跑大量数据,建议先用一两份小样本验证效果再大规模上。

## 四、你接下来真正该做的事:建一个评测集

我上面说的所有"应该改善"都是基于经验的定性预测。**要让这些话变成简历上的数字,你必须自己测**。这也是面试官追问"你怎么知道 v3 比 v2 好"时唯一能站住的答案。

### 最小评测集(2 小时内能搭好)

1. **挑 3–5 段真实(或脱敏 / 虚构)访谈片段**,各 3–5 分钟,覆盖:典型访谈、有跑题内容的访谈、含专业术语的访谈。
2. **手工标注 ground truth**:每段访谈,你人工写下「应该提取的受访者发言 + 应该匹配的题目 + 应该给出的编码」。这一步耗时但不可省。
3. **指标设计**(简单实用,不需要复杂):

   | 任务 | 指标 | 怎么算 |
   |---|---|---|
   | extract_answers | 引用保真率 | (original_text 逐字出现在转写中的条数 / 总条数) |
   | extract_answers | 题目匹配准确率 | (LLM 匹配题目 == 你标的题目)/ 总条数 |
   | extract_answers | OFF_TOPIC 召回率 | (LLM 标 OFF_TOPIC 且你也标 OFF_TOPIC)/ 你标的 OFF_TOPIC 总数 |
   | code_response | 候选码采纳率 | (你打 ✓ 的候选码 / 全部候选码) |
   | code_response | source_quote 保真率 | source_quote 逐字出现在原回答中的比率 |
   | clean_text | 过删率 | (你认为不该删但被删的 token 数 / 总删除数) |

4. **跑 v2 prompts 和 v3 prompts 各一遍,对比指标**。

### 这套评测的价值

- 简历上你能写出**真实数字**:"将引用保真率从 X% 提升至 Y%","候选码采纳率达到 Z%"——这是大厂 AI PM 面试官想看到的东西。
- 之后每改一版 prompt,跑一遍这个评测集,你就有了 **CHANGELOG 里每个版本的量化对比**——这才叫 harness 工程。
- 这个评测集本身也是项目的一部分,可以放进 GitHub repo,放进 README,放进简历。

## 五、建议你顺手做的两件代码侧改动

不是 prompt 的事,但配合 v3 才能发挥全部价值:

1. **加一道 `original_text in transcript` 的校验**:LLM 出错时拒绝那条记录或降分。这是反幻觉的兜底防线。
2. **加一道 `source_quote in answer_text` 的校验**:同理,编码 source_quote 必须能在原回答里找到。

这两个校验加上之后,你的「引用保真率」指标基本可以保证接近 100%——这是个非常实在的工程加分项。
